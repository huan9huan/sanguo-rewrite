# v0.3.3 Content Workflow Refactor

## Goal

这一轮不先追求“功能更多”，而是先把 content build workflow 变成一条更短、更稳、更容易理解的路径。

目标只有三个：

- 内容来源更单一
- build 入口更直接
- website 同时更像读者产品，而不是 repo 可视化器

## Current Friction

当前有几个明显的复杂点：

### 1. Site 有三套内容读取模式

- 直接从 repo 读 `story/` 和 `memory/`
- 读取 `site/public/content/` 导出的 JSON
- 读取 `CONTENT_BASE_URL` 远端内容

这会带来几个问题：

- 心智模型不清楚
- 本地调试路径太绕
- bug 出现时不容易判断是 repo、export、还是 remote 的问题

### 2. `current/` 已经是 handoff surface，但还没有成为真正唯一入口

文档里已经把 `current/` 定义成站点首选来源，但代码还保留了不少 legacy fallback。

这导致：

- site loader 逻辑很长
- passage 状态判断不够直接
- workflow 上“promote to current”这个动作价值不够明确

### 3. export script 和 repo loader 有重复逻辑

`site/lib/repo-content.ts` 和 `site/scripts/export-content.mjs` 里有很多重复：

- frontmatter 解析
- latest version fallback
- image metadata
- scene/review/layout 组装

结果是：

- 改一次 schema，要改两处
- 容易出现 site 页面和导出 JSON 不一致

### 4. Creator Mode 现在更像“把所有内部信息都摊开”

它确实透明，但还不够“产品化”：

- 对第一次来的用户有点重
- “现在该看什么”不够明确
- passage 的可操作状态不够突出

## v0.3.3 Core Decisions

### Decision 1: 定义一个单一内容装配层

新增一个共享 content builder 模块，负责从 repo 组装站点需要的标准数据。

建议形式：

- `site/lib/content-loader/`
- 或 `site/lib/content-source/`

它应该成为唯一负责这些事的地方：

- 读取 `story/`、`memory/`
- 优先读取 `current/`
- 在迁移期处理 legacy fallback
- 组装 `Passage` / `Chapter` / `Book` 数据

然后：

- `repo-content.ts` 只做 repo source adapter
- `export-content.mjs` 复用同一套装配逻辑
- `content.ts` 只负责选择“本地已导出内容”还是“实时 repo 内容”

一句话：不要让 export 再自己重新理解一次 repo。

### Decision 2: 把 `current/` 升级成网站唯一业务入口

迁移期可以保留 fallback，但要明确规则：

- website 展示层只认“标准化后的 current state”
- legacy 文件只在 builder 内部兜底
- 新 workflow 一律围绕“生成 -> review -> promote to current -> site 可见”

这样 `current/` 才真的变成：

- 创作 handoff surface
- site 消费入口
- publish 前 staging area

### Decision 3: dev 模式默认直接读 repo

建议把本地开发默认模式改成：

- `npm run dev` -> Next 直接读 repo

而不是：

- 先 export
- 再起 content server
- 再让 Next 通过 `CONTENT_BASE_URL` 读取

导出 JSON 应该服务于：

- deploy
- 静态内容缓存
- repo 外部托管

而不是本地日常开发。

### Decision 4: export 成为 deployment step，不再是日常心智中心

建议保留：

- `npm run content:export`
- `npm run build:content`

但在产品和代码结构上都把它降级成：

- 发布时快照
- 部署时 artifact

不是平时理解网站内容的第一入口。

### Decision 5: website 收束成两个更清晰的模式

不是继续“堆更多信息”，而是把信息压成更自然的阅读路径。

#### Reader Mode

重点只保留：

- passage 标题
- 可读正文
- 上下节导航
- 漫画入口
- 很轻的 summary / status

读者不需要先理解 spec 才能读。

#### Creator Mode

改成“工作台视图”，优先回答：

- 这节现在处于什么状态
- 当前展示的是哪份 draft / review / comic
- 还缺什么才能进入下一步
- 这节的核心 story intent 是什么

不是把所有内部字段默认摊开，而是先突出 workflow state。

## Suggested Information Model

建议在 builder 层给每个 passage 统一产出一个更明确的状态对象：

```ts
type PassageWorkflowState = {
  has_draft: boolean;
  has_review: boolean;
  has_approved_cn: boolean;
  has_comic_image: boolean;
  has_comic_json: boolean;
  display_status: string;
  reader_text_source: "approved_cn" | "draft_cn" | "none";
};
```

这样页面层不需要自己猜：

- 应该显示 Approved 还是 Draft
- 有没有足够内容进入 reader mode
- creator mode 该突出哪个缺口

## Refactor Plan

### Phase 1: 先统一数据层

- 抽出共享 builder
- 去掉 export 和 repo loader 的重复组装逻辑
- 把 `current/` 读取规则集中到一处

这是最重要的一步。

### Phase 2: 简化本地开发模式

- `site` dev 默认直接读 repo
- 保留 export，但不再让它参与默认本地调试
- 明确 `CONTENT_BASE_URL` 只用于部署或外部内容源

### Phase 3: 重做 Creator Mode 的首页和 passage 卡片

优先展示：

- workflow status
- current assets
- passage goal / conflict / turn
- selected review issues

少一点信息平铺，多一点“下一眼该看哪”。

### Phase 4: Reader Mode 再瘦身

- 更聚焦正文
- 减少 pipeline 术语
- 把“创作过程”留给 creator mode

## Concrete First Tasks

如果这轮我们要真开工，我建议先按这个顺序拆：

1. 抽 `site/lib/content-builder.ts`
2. 让 `export-content.mjs` 调用共享 builder，而不是自己再解析一遍 repo
3. 简化 `site/lib/content.ts` 的模式选择逻辑
4. 把 `npm run dev` 改成 repo-first
5. 重做 Creator 首页，把“workflow state”放到 passage card 第一屏

## Success Criteria

这轮完成后，应该达到：

- 新人能在几分钟内看懂“内容从哪里来”
- 本地开发不需要先理解 export server
- `current/` 的语义更强
- site 更像产品，而不是一个中间构建结果浏览器
- 后续要做 EN、publish、workflow automation 时，不需要再碰三套内容路径

## Short Version

一句话方向：

先把内容层收成“一份 current state + 一个共享 builder + 一种默认 dev 方式”，再去打磨 website 体验。
