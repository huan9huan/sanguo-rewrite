# v0.2.3 Kickoff

## Goal

`v0.2.3` is the first reader-facing website version with a clear product focus:

- one public reading product
- no public creator mode
- passage reading as the main experience
- comic integrated into reading, not separated from it
- content decoupled from the repo and deployable through object storage

This version should make the project easier to read, easier to deploy, and easier to extend.

## Product Position

The public website is a reading website, not a studio dashboard.

Primary audience:

- readers who want to enter the story quickly
- early supporters who want to browse the rewrite

Not primary in `v0.2.3`:

- public process transparency
- deep spec browsing
- review and memory inspection

Those workflow surfaces may return later as an internal or hidden studio route, but they are not part of the main public product in this version.

## Core Decisions

### 1. Reader first

The public site should remove the current dual-mode framing.

Instead of:

- Reader Mode
- Creator Mode

Use:

- Home
- Read

The reading product should feel simple and quiet.

### 2. Passage and comic are one reading unit

The user is not choosing between a prose product and a comic product.

Each passage is presented as a blended reading experience:

- passage text is the main reading spine
- comic supports scene understanding, mood, and pace
- comic can be expanded into a focused view at any time

Default reading state:

- blended passage + comic reading

Optional reading state:

- comic focus view in modal or separate window

### 3. Content and site are decoupled

The website should not depend directly on raw repo structure at runtime.

Instead:

- repo files remain the production source of truth
- a build step exports website-ready content
- the exported content is deployed independently

Deployment targets:

- production: S3 or GCS
- local development: local content website or static content server

## Reader Experience

## Reading principles

- story first
- low friction
- minimal metadata in the main flow
- comic helps reading, never blocks reading

## Reading page structure

Each passage page should include:

1. passage title and light context
2. main reading body
3. integrated comic moments
4. optional comic focus action
5. light navigation to previous and next passage

Information that should be secondary or collapsed:

- review status
- dramatic question
- production labels
- large spec summaries

## Comic integration model

Avoid:

- full comic page first, prose second
- large card stacks with equal visual weight
- creator-facing labels in the reading flow

Prefer:

- prose-first layout
- comic blocks placed at natural story beats
- tap or click to enlarge comic
- optional open-in-new-window comic reader

The default user feeling should be:

"I am reading a story, and the comic is helping me see it more clearly."

## Website Scope

## Public pages in v0.2.3

- `/`
- `/read`
- `/read/[passageId]`

Optional if needed in phase 2:

- `/chapter/[chapterId]`

## Page goals

### Home

Should answer quickly:

- what this project is
- why it exists
- where to start reading

### Read index

Should let readers:

- browse available passages
- see order and light summary
- enter a passage quickly

### Passage page

Should let readers:

- read the passage text cleanly
- use comic as visual support
- move to the next passage without returning to the index

## Content Delivery

## Why decouple content

The repo structure is good for production workflow but not ideal for runtime serving.

Decoupling content gives us:

- simpler site code
- easier CDN delivery
- cheaper static hosting
- cleaner future support for multilingual content

## Proposed website content outputs

### 1. Manifest

A single top-level JSON file for browsing and routing.

Suggested fields:

- project title
- project description
- chapters
- passages
- reading order
- available assets
- timestamps or version markers

### 2. Passage payload

One JSON file per passage.

Suggested fields:

- `id`
- `chapter_id`
- `title`
- `summary`
- `reading_text`
- `approved_cn`
- `draft_fallback`
- `comic`
- `cover_image`
- `prev_passage_id`
- `next_passage_id`

### 3. Comic assets

Per passage comic assets should be exported in website-ready form:

- image URL
- frame layout
- text overlay data if needed
- alt text

## Runtime model

### Production

- content bundle uploaded to S3 or GCS
- site reads manifest and passage payloads from object storage or CDN URLs

### Local

- local content website or static content server
- same manifest and payload shape as production

This keeps local and production behavior aligned.

## Technical Direction

## Frontend direction

The current site already has a strong warm editorial visual base.

For `v0.2.3`, the frontend work should focus on:

- simplifying navigation
- removing creator-facing noise
- improving reading hierarchy
- making blended passage + comic reading feel intentional

Do not spend the phase on:

- overbuilding admin features
- exposing the full pipeline publicly
- adding many separate modes

## Backend/content direction

Add a content export layer between `story/` and `site/`.

This layer should:

- read repo source-of-truth files
- normalize latest readable content
- export website-ready JSON
- support local preview and cloud deployment

## Proposed Phase 1 Deliverables

### Product

- public site positioned as a reading site
- creator mode removed from main public flow
- passage page defined as the core experience

### Content

- manifest schema defined
- passage payload schema defined
- export script or pipeline stub created

### Frontend

- homepage simplified
- `/read` index implemented or refined
- `/read/[passageId]` implemented
- comic focus interaction implemented

### Deployment

- local content serving path works
- cloud storage target prepared for S3 or GCS

## Suggested Development Order

1. lock public IA and page scope
2. define manifest and passage payload shape
3. build content export layer
4. switch frontend data access to exported content
5. implement new read index
6. implement passage reading page
7. add comic focus view
8. prepare object storage deployment path

## Acceptance Criteria

`v0.2.3` is successful if:

- a new reader can understand the project in a few seconds
- the public site no longer feels like a mixed reader/studio tool
- a reader can browse and read passages without production noise
- comic meaningfully supports text reading
- the same content format works locally and in cloud deployment

## Non-goals

Not required in this version:

- full creator workspace on the public site
- full process archive browsing
- editing tools in the website
- full multilingual delivery
- deep search

## Immediate Kickoff

Start implementation with these concrete workstreams:

1. create the exported content contract
2. refactor the site into a single public reading product
3. build the passage reading page with integrated comic support
4. wire local content serving first, then prepare S3/GCS deployment
